Installation Guide

Guppty is a small, indentation-based programming language written entirely in Rust. Install it with one command, write programs in files that end with .gup, compile them to bytecode, and run them on a lightweight virtual machine — all from your terminal.

The installer checks whether Rust is already on your machine. If it is, the script keeps going. If it is not, the script downloads and installs Rust for you, then builds a fresh Guppty CLI onto your PATH so you can create, compile, and run your own programs immediately.

One-line install

macOS and Linux

Recommended

Paste this into your terminal. It performs a fresh install: detects Rust, installs it only when needed, fetches Guppty, builds the guppty command, verifies the install with an example, and prints next steps for writing your own .gup programs.

You only need to run the install command once. After that, guppty stays on your machine and you can create, compile, and run .gup programs anytime.

curl -fsSL https://raw.githubusercontent.com/adrian-1-cardona/guppty/main/install.sh | bash

Windows (PowerShell)

Same fresh install flow

irm https://raw.githubusercontent.com/adrian-1-cardona/guppty/main/install.ps1 | iex

Make your own program

Anytime after that one install

.gup files

After the one-time install, guppty stays on your machine. Create a new program, compile-check it, and run it whenever you want. Every Guppty source file uses the .gup extension.

guppty new hello
guppty compile hello.gup
guppty hello.gup

Project

About Guppty

Guppty is a small, indentation-based programming language that was built from scratch as an educational project in Rust. The core idea behind it is simple: take a plain text file with a .gup extension, turn that text into something a computer can actually execute, and make every step of that transformation visible and understandable. Unlike production languages that optimize for performance and hide their complexity behind layers of abstraction, Guppty is designed to be read, studied, and taken apart. Every component of the language pipeline lives in a single Rust crate with clearly named source files, and the entire project is small enough that a single person can hold the whole thing in their head at once.

The language itself supports the features you would expect from a small scripting language. You can define variables, perform arithmetic, work with strings and booleans, write conditional logic with if and else blocks, iterate with while and for loops, define functions with parameters and return values, and even create closures that capture values from their surrounding scope. The syntax is indentation-based, which means there are no curly braces or semicolons to worry about. You write your code, indent the blocks that belong inside control structures or function definitions, and the language figures out the rest. This makes the source files visually clean and easy to follow, especially for people who are new to programming.

Under the hood, Guppty has two separate execution backends. The primary path, which is the default when you run a program, involves a compiler that transforms the abstract syntax tree into a sequence of bytecode instructions, and a virtual machine that executes those instructions using a stack-based architecture. The secondary path is a tree-walking interpreter that reads the AST directly and evaluates it node by node without any intermediate compilation step. Both backends produce identical results for any correct program, and they exist side by side so that you can compare how the same source code gets handled by two fundamentally different execution strategies. The VM is faster for complex programs and represents the more modern approach to language implementation, while the interpreter is simpler to understand and serves as a reference implementation that you can use to verify the compiler's correctness.

The project also includes a comprehensive set of example programs in the examples directory, each paired with an expected output file that defines exactly what the program should print. These examples serve double duty: they demonstrate language features to new users, and they form the basis of the integration test suite that runs every time you modify the language. This means that if you change something in the lexer, parser, compiler, or VM, and that change breaks the behavior of any example program, the tests will catch it immediately. It is a tight feedback loop that keeps the language honest and makes it safe to experiment with changes.

Performance

VM Benchmarks

One of the most interesting aspects of having two execution backends in the same project is that you can measure them against each other. The benchmark below compares the average end-to-end runtime of the bytecode-compiled VM path against the tree-walking interpreter path, using the recursion_fib.gup example program as the test case. This program computes Fibonacci numbers recursively, which means it generates a significant number of function calls and exercises the call stack heavily, making it a reasonable stress test for both backends.

The numbers you see below come from 200 consecutive runs of the release-built binary on the same machine, measuring the total wall-clock time from process start to process exit. This includes not just the execution of the Guppty program itself, but also process startup, file I/O to read the source file, lexing, parsing, and in the case of the VM path, compilation to bytecode. Because the test program is quite small, process startup and file reading represent a large fraction of the total time, which is why the two backends appear so close to each other in these results.

VM (compile + execute) 2.69 ms
Tree-walker (interpret) 2.64 ms

The key takeaway here is not that one backend is faster than the other. At this scale, the difference is negligible and falls well within the margin of measurement noise. What matters is that both backends are measurable and comparable, which means that as the language grows and programs get more complex, you will be able to see exactly how changes to the compiler or VM affect performance relative to the simpler interpreter baseline. The benchmark infrastructure exists so that future optimizations have a clear before-and-after comparison point.

It is also worth noting that the VM path includes an extra step that the interpreter does not: the compilation phase where the AST gets transformed into bytecode. For tiny programs like this fibonacci example, that extra work is barely noticeable. But for larger programs with many functions and complex control flow, the VM path should eventually pull ahead because it executes precompiled instructions rather than re-interpreting the tree structure on every evaluation. The compilation step is an upfront cost that pays for itself as program complexity increases.

If you want to run these benchmarks yourself, you can build the project in release mode and use a simple timing loop in your shell. The command looks something like running the release binary 200 times against the recursion_fib.gup example and averaging the results. The exact numbers will vary depending on your hardware, but the relative comparison between the two backends should remain consistent across different machines.

Purpose

Why Guppty?

The world does not need another programming language for building production software. There are already excellent tools for every imaginable use case, from systems programming to web development to data science. Guppty exists for a completely different reason: it is a learning tool. It was built to answer the question of what actually happens when you type some text into a file and tell a computer to run it. Most programmers use languages every day without ever looking under the hood, and that is perfectly fine for getting work done. But if you have ever been curious about how a programming language actually works at a fundamental level, from raw text to executed instructions, Guppty gives you a complete and readable implementation to study.

The project is intentionally small. The entire source code fits in a handful of Rust files, and none of them are particularly long or complex. You can read the lexer and understand exactly how characters become tokens. You can read the parser and see how tokens become an abstract syntax tree. You can read the compiler and trace how each AST node maps to a sequence of bytecode instructions. And you can read the VM and watch how those instructions get executed one by one on a simple stack. Every step is transparent, every decision is local, and there is nothing hidden behind layers of abstraction that you would need a PhD in computer science to understand.

This size constraint is not a limitation; it is a deliberate design choice. When a language is small enough to fit in your head entirely, you can make changes to it and immediately understand the consequences. You can add a new keyword to the syntax definition file, update the lexer to recognize it, extend the parser to handle it, teach the compiler to emit bytecode for it, and make the VM execute that bytecode, all in a single afternoon. That kind of immediate, end-to-end feedback is incredibly powerful for building intuition about how languages work. You are not just reading about compiler theory in a textbook; you are actually doing it, with real code that compiles and runs.

Guppty also serves as a personal portfolio piece and a demonstration of what it takes to build a language from nothing. The project shows competence with Rust, understanding of data structures and algorithms, familiarity with testing practices, and the ability to manage a multi-component system where changes in one place ripple through many others. It is the kind of project that demonstrates depth of understanding rather than breadth of feature coverage, and that is exactly what makes it valuable both as a learning experience and as something to share with others.

Under the Hood

Technologies Used

Guppty is built with a deliberately small technology stack. The entire project is written in Rust using the 2021 edition, and it has zero external runtime dependencies. This was a conscious choice: by avoiding external crates for the core language implementation, every line of code in the lexer, parser, compiler, and VM was written by hand and can be read without needing to understand someone else's library. The only tool you need to build and run Guppty is the standard Rust toolchain, which includes the compiler and the Cargo build system.

The language follows a classic pipeline architecture that you would find in any compiler textbook, but implemented in a way that is easy to follow and modify. Source code enters the pipeline as raw text and flows through four distinct stages before producing output. Each stage transforms the data into a progressively more structured form, and each stage is implemented in its own dedicated source file so you can study them in isolation.

The first stage is the lexer, which reads the raw characters of your source file and groups them into meaningful tokens like numbers, strings, identifiers, operators, and keywords. The lexer also handles indentation tracking, which is what allows Guppty to use whitespace for block structure instead of braces. The second stage is the parser, which takes the flat sequence of tokens and builds them into a hierarchical abstract syntax tree that represents the logical structure of your program. The third stage is the compiler, which walks the AST and emits a sequence of bytecode instructions that encode the same logic in a format the VM can execute efficiently. And the final stage is the virtual machine, which reads those bytecode instructions one at a time and executes them on a stack, producing the output your program was designed to generate.

Rust 2021

Rust was chosen as the implementation language because it provides the exact combination of features that a language tool needs. It compiles to a fast native binary that starts up instantly, which means the CLI feels responsive even for trivial programs. Its type system with enums and pattern matching is ideal for representing tokens, AST nodes, and bytecode instructions in a way that the compiler can check for completeness. And Cargo provides a built-in test runner, build system, and dependency manager that makes the development workflow smooth and repeatable across different machines and operating systems.

Stack VM

The stack-based virtual machine is the default execution backend and represents the more sophisticated of the two approaches to running Guppty programs. It works by maintaining a value stack and a call frame stack, pushing and popping values as it executes instructions. This architecture separates the concerns of compilation and execution, which means you can optimize the compiler without touching the VM, or improve the VM without changing how bytecode is generated. It is the same fundamental design used by languages like Java, Python, and Lua in their virtual machines.

Tree Interpreter

The tree-walking interpreter is the alternative execution backend, accessible through the --interp flag. It works by traversing the AST directly, evaluating each node as it encounters it. This approach is simpler to implement and easier to reason about, which makes it a good reference implementation. When you make changes to the compiler or VM and want to verify that the new behavior is correct, you can run the same program through the interpreter and compare the output. If both backends produce the same result, you can be confident that your change is semantically correct.

Integration Tests

The test suite works by running real .gup example programs through the language pipeline and comparing their actual output against expected output files stored in the repository. This approach has several advantages over unit testing individual components: it tests the entire system end-to-end, it uses the same programs that users see in the documentation, and it catches regressions at the behavior level rather than the implementation level. If a test fails, you know exactly what the user would see differently, which makes debugging straightforward.

Notes

What I Learned

Building a programming language from scratch, even a small one, teaches you things that you cannot learn any other way. There is a fundamental difference between reading about how compilers work in a textbook and actually implementing one yourself, feeling the friction of real decisions, debugging real edge cases, and experiencing the satisfaction of watching your language run a program for the first time. The lessons below are the most significant things this project taught me, not just about language implementation specifically, but about software engineering in general.

Language features connect faster than you expect. When you add a single new keyword or operator to the language, that change touches almost every layer of the system. You update the syntax definition, modify the lexer to recognize the new token, extend the parser to handle the new grammar rule, teach the compiler to emit appropriate bytecode for it, and update the VM to execute that bytecode correctly. Then you write an example program that uses the feature, create an expected output file, and verify that both backends produce the same result. What seems like a small addition from the outside turns out to be a journey through the entire codebase, and that is exactly what makes it educational.

Example programs are deceptively powerful. The .gup files in the examples directory look like simple demonstrations for users, but they quietly serve as the foundation of the entire testing strategy. Every time you run the test suite, those examples are compiled and executed, and their output is compared against the expected files. This means that the documentation and the test suite are the same thing, which keeps them permanently synchronized. You cannot break a feature without breaking a test, and you cannot add a test without also creating a demonstration that users can learn from.

Error messages matter more than almost any other feature. A language can have beautiful syntax and elegant semantics, but if it responds to mistakes with cryptic messages or unhelpful line numbers, people will hate using it. Spending time on clear, located error messages with context about what went wrong and where makes the CLI feel like a real tool rather than a frustrating puzzle. It is the difference between "unexpected token" and "expected a closing parenthesis on line 7, but found the end of the line instead." The second version respects the user's time and helps them fix the problem immediately.

Language

Syntax

Indentation based

Guppty uses indentation to define code blocks rather than curly braces, semicolons, or keywords like "end". This design choice was inspired by Python's approach to code structure, where the visual layout of your code directly corresponds to its logical structure. When you write an if statement, the lines that should execute conditionally are simply indented one level deeper than the if keyword itself. When you define a function, the body of that function is everything that is indented beneath the function signature. This makes Guppty code visually clean and forces a consistent formatting style that makes programs easy to read at a glance.

The language supports several fundamental data types: integers, floating-point numbers, strings enclosed in double quotes, single characters enclosed in single quotes, booleans with the values true and false, and arrays for holding ordered collections of values. Variables are created by assignment without any type annotation or declaration keyword. You simply write the name, an equals sign, and the value you want to store, and the language infers the type from whatever you assign to it. This keeps the syntax minimal and reduces the ceremony needed to get things done.

Below you can explore the different syntax categories by clicking through the tabs. Each one shows a small code example that demonstrates a particular feature of the language, from basic variable assignment and output to control flow, function definitions, closures, and comments. These examples are all runnable programs, meaning you can save any of them to a .gup file and execute them with the guppty command to see the output.

Variables, values, and output

The out() function is how you print values to the terminal. You can pass it any expression and it will display the result. Variables are created simply by assigning a value to a name. There is no let, var, or const keyword needed. The language supports integers, floating-point numbers, strings in double quotes, single characters in single quotes, and boolean values. You can perform arithmetic on numbers and concatenate strings, and out() will display whatever the final evaluated result is.

x = 5
name = "Guppty"
ready = true
letter = 'g'
out(name)
out(x + 10)

If, while, and for

Control flow in Guppty works through indentation. When you write an if statement, the condition goes on the same line as the if keyword, and the code that should run when the condition is true is indented below it. The else keyword sits at the same indentation level as the if, and its body is indented beneath it in the same way. For loops use the range() function with the "through" keyword to define the iteration bounds, where range(1 through 3) will iterate with i taking the values 1, 2, and 3. While loops work the same way: the condition is on the while line, and the loop body is indented beneath it.

score = 85
if score >= 80
    out("You passed!")
else
    out("Keep studying!")

for i in range(1 through 3)
    out(i)

Functions and returns

Functions in Guppty are defined by writing the function name followed by its parameters in parentheses, with the function body indented on the following lines. There is no "def" or "func" keyword. The language recognizes a function definition by seeing an identifier followed by parentheses at the start of a line. Return values are sent back using the return keyword. You can call functions by name and pass arguments just like you would in any other language. Functions can call other functions, and they can call themselves recursively.

add(a, b)
    return a + b

out(add(2, 3))

Closures

Closures are functions that capture and remember values from the scope in which they were defined, even after that scope has finished executing. In Guppty, you create a closure by defining a function inside another function and then returning it. The inner function has access to the outer function's variables, and those variables remain alive as long as the closure exists. This is a powerful pattern that enables things like factory functions, stateful callbacks, and encapsulated counters. The example below shows a makeAdder function that creates and returns a closure which remembers the value of x from its parent scope.

makeAdder(x)
    addIt(y)
        return x + y
    return addIt

adder = makeAdder(5)
out(adder(3))

Comments

Comments in Guppty use the double-slash syntax that will be familiar to anyone who has used C, Java, JavaScript, or Rust. A comment starts with two forward slashes and continues to the end of the line. Everything after the double slash is ignored by the lexer and does not affect the behavior of your program. Comments can appear on their own line, or they can be placed after a statement on the same line. They are useful for leaving notes to yourself or others about what your code does, why a particular approach was chosen, or what you plan to change in the future.

// this is a comment
out(42) // this is an inline comment

Programs

Examples

The examples directory is the best place to start when you want to understand what Guppty code looks like in practice. Every file in that directory is a complete, runnable program that demonstrates one or more language features. The programs are deliberately small, usually just a few lines, so you can read them in seconds and understand exactly what they do. Each example has a corresponding file in the examples/expected subdirectory that contains the exact output the program should produce, which means you can verify that your installation is working correctly by comparing the actual output against the expected output.

The examples below show three different programs that represent increasing levels of language complexity. The hello world example is the simplest possible Guppty program, just a single line that prints a string. The recursion example shows how functions can call themselves to solve problems that have a naturally recursive structure. And the closure counter example demonstrates one of the more advanced features of the language: the ability to create functions that maintain their own private state through captured variables. Together, these three programs give you a good sense of the range of things Guppty can express, from trivial one-liners to patterns that would be right at home in a production codebase.

Hello World

The simplest Guppty program. A single call to out() with a string argument prints that string to the terminal and exits. This is the first program most people write when learning a new language, and in Guppty it could not be any shorter.

out("Hello World!")
guppty examples/hello.gup

Recursion

This program defines a factorial function that calls itself with a smaller argument until it reaches the base case. Recursion is fully supported in both the VM and interpreter backends, including deep call stacks and proper return value propagation.

factorial(n)
    if n <= 1
        return 1
    return n * factorial(n - 1)

out(factorial(5))
guppty examples/recursion_factorial.gup

Closure Counter

This example creates a counter function that maintains its own internal state. Each time you call the returned increment function, it updates and returns the count. The captured count variable lives on even after makeCounter has returned, which is the essence of how closures work.

makeCounter(start)
    count = start
    increment()
        count = count + 1
        return count
    return increment
guppty examples/closure_counter.gup

Community

Contribute

Guppty is an open-source project that welcomes contributions from anyone who finds it interesting, regardless of experience level. Whether you want to fix a bug, add a new language feature, improve error messages, write documentation, or simply run the test suite and report what happens, your involvement makes the project better. The codebase is intentionally small and well-organized, which means getting oriented does not take long. You can read through the main source files in an afternoon and have a solid understanding of how everything fits together.

The contribution workflow follows standard GitHub practices. Fork the repository, create a branch for your change, make your modifications, run the test suite to make sure nothing is broken, and open a pull request with a clear description of what you changed and why. Small, focused pull requests are much easier to review than large ones that touch many files at once, so try to keep each pull request focused on a single concern. If you are adding a new language feature, include a new example .gup file that demonstrates the feature and a matching expected output file so it becomes part of the test suite automatically.

There are a few guidelines worth keeping in mind when you contribute. First, always run cargo test before opening a pull request. The test suite is the single source of truth for whether the language still behaves correctly, and it catches most regressions immediately. Second, if you are adding or modifying syntax keywords, keep them centralized in src/syntax.rs rather than scattering string literals across multiple files. This makes it easy to see all the reserved words in one place and reduces the chance of inconsistencies. Third, make sure you are not accidentally committing the target directory, which contains compiled build artifacts that do not belong in the repository. The .gitignore file should handle this automatically, but it is worth double-checking if you are new to Rust projects.

If you are unsure where to start, look at the existing example programs and try writing a new one that exercises a combination of features in an interesting way. Or pick one of the error messages in the language and see if you can make it clearer or more helpful. Or look at the issues tab on GitHub to see if there are any open items that match your interests. Every contribution, no matter how small, helps the project grow and improves the experience for everyone who uses it.

Troubleshooting

Common Issues

When you are working with a project that builds from source and runs on multiple operating systems, you will occasionally run into setup problems. Most of these issues have straightforward solutions, and nearly all of them come down to either a missing tool on your PATH or a misunderstanding about how the build process works. Below are the most common problems people encounter when getting started with Guppty, along with detailed explanations of what causes each one and how to fix it.

The one-line installer failed. Make sure you have curl (macOS/Linux) or can run PowerShell (Windows). The script needs network access to download Rustup and the Guppty source. If Rust was just installed, open a new terminal and re-run the installer so PATH picks up ~/.cargo/bin.

cargo or rustc is not found. The one-line installer installs Rust for you when it is missing. If you are installing manually, use rustup.rs and then restart your terminal. On macOS and Linux, the installer adds a line to your shell profile that puts ~/.cargo/bin on the PATH, but that change only takes effect in new terminal sessions. On Windows, you may need to log out and back in for the PATH change to propagate system-wide.

guppty is not found after install. This typically means the binary is in a directory that your shell does not know about. The binary gets placed in ~/.cargo/bin on macOS and Linux, or in C:\Users\YourName\.cargo\bin on Windows. Verify that this directory is on your PATH by running echo $PATH on macOS/Linux or echo %PATH% on Windows and looking for the cargo bin entry. If it is missing, add source "$HOME/.cargo/env" to your shell profile, or re-run the one-line installer after opening a fresh terminal.

The CLI cannot read my file. This error appears when you pass a file path that does not exist. Create one with guppty new my_program (that writes my_program.gup), or pass the full path like guppty ./my_program.gup. On Windows you can use either forward slashes or backslashes.

Indented blocks behave unexpectedly. Guppty uses indentation to determine block structure, which means that inconsistent spacing can cause the parser to misinterpret your code. Make sure you are using the same indentation style throughout your file. Spaces and tabs should not be mixed. The convention in Guppty examples is four spaces per indent level. If you are using a text editor, configure it to insert spaces when you press the tab key and set the tab width to four. The most common mistake is accidentally using a different number of spaces for one line in a block, which makes the parser think the block has ended when it has not.

How do I check that everything still works? Run guppty compile hello.gup on a program you created, or run cargo test from a source checkout. The test suite executes every example in the examples directory and compares the output against the corresponding expected output file.

Updates

Release Notes

Guppty uses semantic versioning to track its progress, with the version number defined in the Cargo.toml file at the root of the repository. Each release represents a stable snapshot of the language that passes all tests and can be installed and used reliably. Below are the current release and planned future milestones for the project. As new features are added, new versions will appear here with descriptions of what changed and what you can expect from each update.

Latest

One-line install + your own .gup programs

Fresh install from a single terminal command on macOS/Linux (install.sh) and Windows PowerShell (install.ps1). The installer checks for Rust, installs it when missing, builds the Guppty CLI, and verifies the setup. The CLI now includes guppty new, guppty compile, guppty run, and guppty help so anyone can create, compile-check, and run their own .gup programs after downloading Guppty onto their computer.

Foundation

Version 0.1.0

Initial release of Guppty with the VM and tree-walking interpreter backends, core language features (variables, arithmetic, strings, booleans, conditionals, loops, functions, closures), example programs with expected outputs, and an integration test suite.

Coming Soon

Binary Downloads

A future release will add prebuilt binaries for macOS, Linux, and Windows through GitHub Releases so users who do not want to compile from source can download a single executable. The one-line installer already automates the from-source path today.
